Token Economy – Operating Model & Requirements

Functional and conceptual requirements for how the token economy lives inside the telco Benefit Wallet.

1. Core principles

The token economy turns every interaction into value and uses the Benefit Wallet as a single place where customers can see, understand and use their rewards.

  • One universal token powers all reward mechanics.
  • Every interaction can generate value: monetary, behavioural, or loyalty-based.
  • The wallet acts as a persistent balance plus a library of perks and benefits.
  • Benefits can be:
    • Direct – earned as tokens immediately.
    • Claimable – unlocked as a benefit card that must be claimed.
  • Each benefit has a clear lifecycle: created → visible → claimed → redeemed → expired.

2. Token structure and transaction types

2.1 Token types

Standard tokens Bonus tokens Locked tokens

  • Standard tokens – core currency, used for most redemptions.
  • Bonus tokens – campaign or seasonal multipliers with shorter validity.
  • Locked tokens – count for tier progression but cannot be spent.

2.2 Transaction types – earning

  • Billing: bill payment, autopay enrollment, on-time payment streaks.
  • Top-ups: prepaid recharges and bundles.
  • Devices: new device, upgrades, trade-in deals.
  • Accessories: single or bundled purchases.
  • Add-ons: roaming passes, data add-ons, cloud storage.
  • Tenure & value: months in network, ARPU stability, multi-service bundles.
  • Engagement: app missions, profile completion, surveys, digital self-care flows.
  • Referrals: invited customers who activate and pay.

2.3 Transaction types – spending

  • Telco benefits: bill discounts, data, roaming, OTT, cloud, insurance, device subsidies.
  • Partner benefits: gyms, travel, retail, dining, entertainment.
  • Perks: trade-in boosts, accessory vouchers, extended warranty, priority support.
  • Gifting: send tokens or specific perks to other customers.

3. Earning logic – direct vs claimable

3.1 Direct acquisition (immediate tokens)

Used for simple, frequent and emotionally relevant actions.

  • Bill payment (especially via digital channels).
  • Prepaid top-ups, add-on purchases and roaming passes.
  • Standard device and accessory purchases.
  • Daily / weekly missions and app engagement.
  • Referrals that lead to activation and first payment.

Requirement: every such action must produce a deterministic token amount based on a configured rule set (percentage, fixed value, or tier-based multiplier).

3.2 Claimable benefits (conditional perks)

Used when we want to control cost exposure, encourage repeat behaviour and drive wallet usage.

  • Example: buy a specific device → unlock a future trade-in boost.
  • Example: buy 3 accessories within 9 months → unlock -20% accessory voucher.
  • Example: take device insurance → unlock one-time screen repair voucher.

Requirement: system must be able to create a “benefit card” in the wallet with:

  • Eligibility rule (what had to happen).
  • Activation rule (what happens when user claims it).
  • Validity period (from–to dates).
  • Links to specific products / journeys (e.g. device checkout).

4. Token valuation model

Tokens must be easy to understand for customers and easy to manage for Finance.

  • Define a base internal reference: e.g. 100 tokens ≈ 1 BGN value.
  • Define earn rates by category:
    • Bill payment: 1–3% “back” in tokens.
    • Device purchase: higher multiplier (e.g. 3–5%).
    • Add-ons: fixed token packages or boosters.
    • Referrals: flat reward (e.g. 500–1000 tokens).
    • Missions: small micro-rewards (5–20 tokens).
  • Define burn rules:
    • Telco benefits: fixed token prices for key items.
    • Partner benefits: negotiated ranges and caps.
    • Device discounts: capped token usage per transaction and per month.

5. Benefit lifecycle

5.1 Stages

  • Creation: triggered by a transaction, campaign or system rule.
  • Notification: push, in-app, or wallet badge.
  • Visibility: appears in wallet under “Available”, “Upcoming” or “Expiring”.
  • Claim: user actively claims, moving benefit into “Active” state.
  • Redemption: applied during a purchase, bill, or add-on flow.
  • Expiration: unused benefits expire, with optional reminder before expiry.

5.2 Required attributes per benefit

  • Unique ID and type (token redemption, voucher, perk, partner offer).
  • Source (which rule, campaign, or transaction generated it).
  • Value definition (fixed discount, % discount, free item, extra data, etc.).
  • Token cost (if any) and caps.
  • Start and end date, grace period if applicable.
  • Eligibility segment (all base, specific plans, specific devices, etc.).

6. Governance & risk control

  • Token budget: quarterly cap on total tokens issued per category.
  • Burn controls: limits on monthly token redemption per user and per product.
  • Anti-gaming: caps on mission frequency, referral validation, fraud checks.
  • Expiry policy:
    • Standard tokens: 12–24 months.
    • Bonus tokens: 30–90 days.
    • Claimable benefits: 14–60 days.
  • Analytics: KPIs (engagement, redemption rate, incremental revenue, retention lift).

7. Long-term evolution model

  • Phase 1 – Closed telco ecosystem: tokens used only for telco benefits.
  • Phase 2 – Partner marketplace: redemption extended to external brands.
  • Phase 3 – Cross-customer flows: gifting, trading, family / household pooling.
  • Phase 4 – Semi-open asset: optional evolution into a broader digital asset model.
